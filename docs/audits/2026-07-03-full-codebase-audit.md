# Full Codebase Audit — 2026-07-03

A read-only audit of the Lifegroups admin operating system (Next.js 16 / React
19 / TypeScript / Supabase), executed per
[`docs/plans/REPO_WIDE_AUDIT_PLAN.md`](../plans/REPO_WIDE_AUDIT_PLAN.md).
Scope: **Security & RLS**, **Architecture & code health**, **Tests & CI**, and
**Docs & domain-language drift**. Successor to
[`2026-06-21-full-codebase-audit.md`](./2026-06-21-full-codebase-audit.md)
(the "prior audit" throughout).

This is a **report-only** pass — no code was changed, no issues were filed.
Every recommendation below is advisory. The only file changes in this branch
are this report and the pointer/blurb lines in `docs/README.md`.

---

## 1. Executive summary

**Overall health: strong, and measurably stronger than 2026-06-21.** Of the
prior audit's 29 findings, **24 are closed, 3 partially closed, and only 2
still open** (§4) — almost all actioned by PR #789 and ADR 0033. The
review-only security guardrails the prior audit worried about are now gating
fitness checks; the DB/RPC layer is byte-identical to what the prior audit
verified (`git diff` over `supabase/` since then is empty) and re-checked
clean. The audit found **no open P0/critical issues** and **no security
holes**: no cross-tier leakage, no exposure of the Private Care Note or
`admin_private_note`, no loosened `USING` clause anywhere in the 128
migrations, zero write RLS policies.

The genuine findings concentrate in three places:

1. **The two surviving P1 carry-overs** — the RLS integration harness is still
   not a _required_ status check (TEST-1), and the core write flows still have
   no end-to-end coverage (TEST-2).
2. **A glossary that lost a model change.** The cell → group-type collapse
   (migration `20260708000000`, #738) landed without an ADR, and `CONTEXT.md`'s
   entire Multiplication block, `README.md`'s Multiply row, and
   `PRODUCT_DEFINITION.md` still teach the retired Audience × Category "cell"
   model — including a dropped table (DOC-1/3/4/11). `CONTEXT.md` also still
   says the Leader surface is gated off, contradicting ADR 0024 (DOC-2).
3. **Spec-doc drift on the security matrix itself** — `RLS_VISIBILITY.md`
   under-counts the enforced matrix (53 vs 56 tables) and omits two
   newer tables whose policies are correct and machine-checked (SEC-1).

## 2. Methodology & scope

Executed per the plan: pre-reading (README, CONTEXT, CLAUDE, AGENTS,
RLS_VISIBILITY, DATA_CLASSIFICATION + manifest, the prior report, ADR 0033),
then a green-baseline check, then **Step 0** (re-verify all 29 prior findings,
§4), then **four parallel read-only passes** (one per dimension), each
returning a findings table; results were synthesised and every load-bearing
claim **spot-verified by reading the cited files directly** before inclusion.

Corrections made during verification are noted inline, as the prior report
does: one pass mis-cited `leader-grade-actions.ts` as living under
`group-health/` (it is `app/(protected)/admin/shepherd-care/leader-grade-actions.ts`);
one pass counted 25 a11y specs (there are 24 `.spec.ts` files); one pass
summarised the prior-finding tally as 25/2/2 (the per-row statuses sum to
24 closed / 3 partially closed / 2 still open); and the sweep-test table count
was re-derived directly (56 distinct tables).

**Baseline:** `npm run test:run` is green at audit time — **347 test files,
3785 tests passing, 1 skipped**. (Prior audit: 311 files / 3552 passing.)

**Severity rubric** (unchanged from the prior audit)

- **P0 / Critical** — violates a documented security invariant or exposes
  sensitive data (private Care Note, `admin_private_note`, cross-tier leakage).
- **P1 / High** — a real correctness/security risk, or visible drift against a
  ratified decision, or a missing guardrail on an invariant that today relies
  only on review.
- **P2 / Medium** — maintainability, coverage gaps, drift that will bite later.
- **P3 / Low** — cosmetic, stylistic, or nice-to-have.

**Out of scope** (unchanged): code fixes/refactors, filing issues,
dependency/CVE scanning, and performance/accessibility deep-dives. No P0
surfaced incidentally in those areas.

**Machine-checked invariants (not re-derived).** The gating fitness suite —
**15 checks** in `tests/fitness/`, confirmed by the green baseline — proves on
every build: `actions-use-run-action`, `audit-no-sensitive-plaintext`,
`care-note-visibility-divergence`, `detail-reads-use-read-batch`,
`leader-allowlist-no-admin-private`, `no-broad-rls`, `no-direct-table-writes`,
`no-hard-delete`, `no-hardcoded-identity`, `no-select-star`,
`no-sensitive-data-in-logs`, `no-service-role`, `rls-coverage-completeness`,
`security-definer-search-path`, and `write-rpc-audit-pairing`. Its support
utilities are themselves tested (`tests/fitness/support/__tests__/`). The
gating lane additionally pins the full per-table visibility matrix (56 tables)
in `lib/admin/__tests__/admin-rls-visibility-sweep.test.ts`. These are cited,
not re-audited; the passes spent their effort on the seams the static scans
cannot cover (RLS `USING` semantics, audit-pairing semantics, revalidate
coverage, doc accuracy).

**Confirmed clear (no findings):**

- **Writes are RPC-only, still.** Zero INSERT/UPDATE/DELETE policies and zero
  `WITH CHECK` clauses across all 128 migrations; the only non-SELECT grant is
  `grant select on invitations to service_role` (`20260607130000`, required by
  the redeem-invite Edge Function, spec-consistent).
- **Super-Admin-only tables all gate correctly** on
  `auth_role() = 'super_admin'` — verified for `audit_events`,
  `audit_events_archive`, `platform_config`, `usage_events`, `invitations`,
  `tombstones`, all three reset-snapshot tables, and
  `account_deletion_requests`. None has a write policy.
- **The two opposite care-note conventions hold, and no new care-adjacent
  table copied the wrong one.** `care_notes` / `prayer_requests`
  (`…author_or_granted_select`): author-always OR (`auth_is_admin()` AND an
  active transparency grant) — the admin arm is never bare and the Super Admin
  is gated on the _same_ grant as the Ministry Admin. SC.4 private notes
  (`20260529008000:112–126`): creator-scoped
  `ministry_admin AND created_by = auth_profile_id()`, Super Admin excluded.
- **`admin_private_note` unreachable from leader routes** (type + allowlist +
  fitness check), and **`staff_viewer` gained nothing**:
  `auth_is_staff_viewer()` is neutralized to `select false`
  (`20260531140000`); no runtime code references the role.
- **RLS is fail-closed at the schema level**: every created table has RLS
  enabled, plus the auto-enable event trigger
  (`20260630000000_db_hygiene_capture_rls_auto_enable.sql`). Helper functions
  were only ever _narrowed_ across redefinitions; no hardcoded UUIDs/emails
  anywhere in policies.
- **Auth parallelization did not weaken the revocation gate.**
  `lib/auth/session.ts`: `getUser()` and the `profiles` read run concurrently,
  but the profile result is consumed only _after_ the
  `if (!user) return anonymous` gate — a revoked user's read is discarded
  server-side. Middleware performs no authorization; `SESSION_PROFILE_COLUMNS`
  keeps the hottest read on a pinned named-column allowlist.
- **Prior SEC-3 fixed:** `redeem-invite` now pads every business-logic return
  to `INVITE_TIMING_FLOOR_MS` (shared `_shared/timing.ts`), and `mapRpcToken`
  collapses distinguishing tokens to a generic `email_unavailable` — timing
  _and_ content enumeration mitigated. `invite-user` still pads; service-role
  keys read only from Deno env in all three Edge Functions; no raw email/token
  in logs or responses.
- **Audit pairing beyond the static check:** all 3 `AUDIT_EXEMPT_WRITES`
  exemptions (`log_usage_event`, `check_invite_redeem_rate`,
  `super_admin_clean_slate_restore_payload`) remain justified,
  machine-enforced (delegation-caller and no-app-grant premises are themselves
  tested), and each carries a dedicated migration test. All 10 write-RPC
  migrations landed since the prior audit (`20260622`→`20260712`) insert the
  audit row in the same function body/transaction, derive the actor via
  `auth_profile_id()`, and use presence flags for sealed content — zero
  sealed-plaintext leaks.
- **Public surface is clean:** the old preview routes are gone;
  `/a11y-harness` is env-gated, force-static, with a type-only Supabase
  import; demo fixtures use only `example.com`/`example.test` identities;
  `/api/vitals` normalizes routes and logs no PII. The admin shell's streamed
  nav kept the guard in front: `admin/layout.tsx` awaits `requireAdmin()`
  before rendering; only the UX-level nav-visibility promise streams, and
  off-nav surfaces retain their own route guards.
