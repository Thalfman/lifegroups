Problem.

Problem.

  • Refactoring RLS stops breaking a regex
  • Directly guards a P0 security invariant
Touches the SEC-1 fitness pin — the existing check stays until the differential test proves stronger; this deepens the guard, it does not relax it.

Problem.
The interface of each big shell is its rendered DOM — you can only test past it. Fifteen shells hold the app’s stateful wiring with no unit locality, while their pure data siblings are all tested.
Solution.
Not a blanket refactor: shell by shell, where wiring is branchy, pull the state choreography into a view-model module the shell binds — starting with the check-in twins and the invite workflow.
  • Locality: wiring bugs get a unit test target
  • Check-in twins collapse to one view model
  • Speculative: apply only where branching earns it
Top recommendation

1 · Grow the Write Action Runner three small seams

Highest leverage per line changed: three optional interface members collapse ~450 lines of hand-rolled, untested write ceremony on the most security-sensitive writes in the app (invites, test accounts, account deletion), shrink the fitness suite’s EXEMPT list, and stop RPC error tokens leaking across the seam — all inside the runner ADR 0001 already established, with its existing test suite covering every migrated action for free. Candidates 2 and 3 are equally safe and make good follow-ups.
